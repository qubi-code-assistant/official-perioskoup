# Perioskoup Landing Page ‚Äî Design & UX Audit
**Date:** 2026-03-19  
**Auditor:** Design & UX Subagent  
**Launch target:** 2026-03-30 (11 days out)

---

## Executive Summary

Perioskoup's landing page is **genuinely strong** for a pre-revenue dental SaaS product. The dark navy brand system is coherent, the EFP award is used as effective social proof, and the dual-audience (patients vs. dentists) architecture is clear. The biggest risks before launch are: a **duplicate-text bug on the Home page**, **no real form submission** on the Waitlist, weak mobile readability for Dongle headings below 360px, and a handful of missing `prefers-reduced-motion` guards.

## 2. Component Quality

### shadcn/ui Usage

The project imports many shadcn/ui primitives but the page code primarily uses **custom HTML elements** with inline styles rather than shadcn components. This is a deliberate design decision (custom `.btn-primary`, `.card`, `.p-input` CSS classes) and is **not wrong** ‚Äî but it means the shadcn components exist without being used in most pages.

**Issues:**
1. **`client/src/components/ui/`** ‚Äî Contains 50+ UI components. None are imported in the 5 main landing pages. If this project ever adds an admin panel or form-heavy page, these will be useful. For the landing page, the custom CSS system is fine. No bug here, but dead weight in the bundle. Consider running `vite-plugin-treeshake` or at minimum checking that unused shadcn components are not included in the production bundle.
2. **`Features.tsx:130`** ‚Äî `dangerouslySetInnerHTML={{ __html: f.desc }}` is used for the Progress Tracking card description (which includes a `<a href='...' style='...'>`). This is an XSS risk if the FEATURES array is ever sourced externally. For a static constant it's fine but worth noting for when CMS integration happens.

### CSS Classes
3. **`index.css:~605`** ‚Äî `.btn-primary` (visible in CSS): check that `min-height: 44px` is set for touch target compliance. The hero CTA uses `padding: "16px 32px"` which should achieve 44px+ height, but the Navbar CTA uses `padding: '9px 20px'` ‚Äî that's approximately 37px height. **Touch target too small on mobile.** Fix: Add `min-height: 44px` to `.btn-primary` with `align-items: center`.
4. **`index.css`** ‚Äî `.section` and `.section-sm` classes are used in `Features.tsx` (`.section`, `.section-sm`) but are not visible in the first 600 lines of CSS. Confirm these are defined in the Tailwind `@layer components` block or Features layout will be broken.

## 3. Mobile Responsiveness

### Navbar
- **`Navbar.tsx:78`** ‚Äî Mobile hamburger is `44√ó44px` ‚úÖ correct touch target.
- **`Navbar.tsx:130`** ‚Äî Mobile drawer uses `fontFamily: 'Dongle', fontSize: 'clamp(36px, 10vw, 48px)'` ‚Äî at 320px viewport this is 32px, which is fine. At 360px it's 36px. ‚úÖ

**Issues:**
1. **`Navbar.tsx:85`** ‚Äî Desktop nav CTA `padding: '9px 20px'` ‚âà 37px height. On tablet (768px) the desktop nav renders ‚Äî this CTA is below the 44px WCAG touch target. Fix: `padding: '11px 20px'` or add `minHeight: '44px'`.

### Home.tsx
2. **`Home.tsx:116`** ‚Äî Hero grid: `grid-cols-1 lg:grid-cols-[1.1fr_0.9fr]`. On tablet (md: 768px‚Äì1023px) the layout is single-column with a full-width phone mockup. The phone mockup max-width is `sm:max-w-[340px]` which is correct. However the phone mockup appears **below** the hero text on mobile/tablet ‚Äî this means the phone is below the fold on small screens. Consider switching to `md:grid-cols-[1.1fr_0.9fr]` if the phone should be visible above-fold on tablet.
3. **`Home.tsx:199`** ‚Äî Features bento grid: `grid-cols-1 md:grid-cols-3`. The span-2 cards (`md:col-span-2`) and span-1 cards create a 2+1 / 1+2 pattern on desktop. On mobile all 4 cards stack full-width ‚Äî ‚úÖ correct.
4. **`Home.tsx:264`** ‚Äî How It Works circles: On mobile the `grid-cols-1` stacks circles vertically. The offset (`md:pt-20`) only applies at md+, so mobile layout is clean. The SVG curve path `<svg className="hidden md:block">` is correctly hidden on mobile ‚úÖ.

### ForDentists.tsx
5. **`ForDentists.tsx:228`** ‚Äî Stats section: `grid-cols-1 sm:grid-cols-3 gap-px bg-[#234966] rounded-2xl overflow-hidden`. The `gap-px` creates 1px dividers between cells. On mobile (grid-cols-1), this creates horizontal dividers between vertically stacked stats ‚Äî acceptable but slightly awkward as the lime separator is very thin. Consider `gap-0 divide-y divide-[#234966]` for mobile and `divide-x` for sm+ using responsive variants.

### Features.tsx
6. **`Features.tsx:126`** ‚Äî Feature grid: `grid-cols-1 sm:grid-cols-2 xl:grid-cols-3`. At 768‚Äì1279px this renders a 2-column grid. ‚úÖ reasonable.

### Waitlist.tsx
7. **`Waitlist.tsx:82`** ‚Äî Role selector: `grid-cols-1 sm:grid-cols-2 gap-3`. On mobile (375px) the two role buttons stack vertically ‚Äî fine. Each button has `padding: 20` which is ~40px height. Acceptable but add `minHeight: 64px` for easier touch.
8. **`Waitlist.tsx:92`** ‚Äî Form inputs use `className="p-input"`. The `.p-input` class must be checked for `font-size: 16px` ‚Äî iOS Safari zooms the page when an input has `font-size < 16px`. If `.p-input` uses 14px font-size, **all iOS users get an unwanted zoom on focus**. Critical mobile fix.

## 4. Animation Quality

### Inventory of Animations
| Animation | CSS Class | Purpose | Performance |
|---|---|---|---|
| Ken Burns on hero bg | `.ken-burns` | Depth/life to hero | ‚úÖ `transform` only, GPU |
| Phone float | `.phone-float` | Draws eye to product | ‚úÖ `transform: translateY` |
| Drift-X on features bg | `.drift-x` | Ambient motion | ‚úÖ GPU |
| Ticker marquee | `.ticker-track` | Brand messaging loop | ‚öÝÔ∏è See issue 1 |
| Reveal fade-up | `.reveal` | Scroll entry | ‚úÖ Intersection Observer |
| Circle pulse (How It Works) | `.circle-pulse` | Step emphasis | ‚öÝÔ∏è See issue 2 |
| CTA orbs (5 total) | `.cta-orb--1` through `--5` | Ambient depth | ‚öÝÔ∏è See issue 3 |
| Grid pulse | `.cta-grid` | Texture | ‚úÖ opacity only |
| Particle rise | `.cta-particle` | Decorative | ‚öÝÔ∏è See issue 4 |
| Scan line | `.cta-scan-line` | Sci-fi texture | ‚öÝÔ∏è See issue 4 |

**Issues:**
1. **`index.css:~390`** ‚Äî `.ticker-track` animation: The ticker uses two identical copies of the text array (`[...Array(2)]`) to create a seamless loop. The `ticker` keyframe goes from `0` to `-50%`. This is correct. However, the ticker is marked `aria-hidden="true"` but the `div` containing it has no `role` and the text appears in the DOM ‚Äî screen readers in some modes may still read it. Also: **no `prefers-reduced-motion` guard** on the ticker animation. Add:
   ```css
   @media (prefers-reduced-motion: reduce) {
     .ticker-track { animation: none; }
   }
   ```
2. **`index.css:~330`** ‚Äî `@keyframes pulse-ring` is defined for `.circle-pulse` (How It Works step circles). The pulse uses a `::before` pseudo-element. However in `Home.tsx:276`, the `.circle-pulse` div has `animationDelay: \`${i * 0.8}s\`` set inline ‚Äî this works but the `circle-pulse` class is not visible in the audited CSS portion. Confirm the CSS pseudo-element pulse is defined correctly.
3. **`index.css:~420-450`** ‚Äî CTA orbs: 5 large blurred elements (`filter: blur(80px)`) animating simultaneously. On mid-range Android (Pixel 6a class), `filter: blur()` on elements > 400px can cause compositing issues. The code comments explicitly note that `will-change` was omitted to save GPU memory ‚Äî this is the right call. However, **the orbs are NOT guarded by `prefers-reduced-motion`**. Add:
   ```css
   @media (prefers-reduced-motion: reduce) {
     .cta-orb { animation: none; }
     .cta-particle { animation: none; display: none; }
     .cta-scan-line { animation: none; display: none; }
   }
   ```
4. **`index.css`** ‚Äî `particle-rise` and `scan-line` animations appear only on the CTA section (not yet visible in audited pages ‚Äî these may be on a CTA-specific page). Both are decorative with no information value. They must have `prefers-reduced-motion` guards.
5. **`Home.tsx:8`** ‚Äî Comment at top correctly documents the animation split fix (reveal-scale outer, phone-float inner). ‚úÖ This was a real bug that was properly fixed.
6. **`Home.tsx:46`** ‚Äî `useReveal(0.12)` ‚Äî the 0.12 threshold means 12% of the element must be visible before the reveal fires. This is appropriate for most elements but on mobile, large sections may trigger reveals awkwardly. Test on 375px to ensure the hero CTA is visible without scrolling.

**Overall animation verdict:** The animation system is purposeful and well-structured. The main risks are `prefers-reduced-motion` non-compliance (a WCAG 2.3.3 issue) and the blur-heavy CTA orbs on low-end mobile. Not gratuitous ‚Äî the Ken Burns, phone float, and ticker all serve the brand.

## 6. Competitive Positioning

### vs. Dentally (UK)
- Dentally: white/clinical design, feature-heavy nav, no consumer-facing patient story
- Perioskoup: **wins on emotional narrative** ("Between visits, we take over"), dual-audience framing, award proof
- Gap: Dentally has integration logos (NHS, practice management systems). Perioskoup has none. **Add 2‚Äì3 integration or platform logos** (iOS App Store, Android, GDPR badge) to the Waitlist or Home page to signal ecosystem legitimacy.

### vs. CareStack (US)
- CareStack: complex PMS with pricing tiers, demo-first CTAs, very enterprise feel
- Perioskoup: **wins on simplicity and patient-centricity** ‚Äî CareStack never mentions patients as end users
- Gap: CareStack shows real product screenshots. Perioskoup's `AppScreenshot` on ForDentists uses a single dashboard image. **The app is not yet built** (implied by beta/coming-soon badges) ‚Äî this is honest but the lack of real UI creates a trust gap. The coded PhoneMockup in the hero partially compensates.

### vs. Dental Intelligence (US)
- DI: analytics-first, revenue-focused for practice owners, heavy data visualization
- Perioskoup: different category entirely (patient behavior, not practice analytics)
- Gap: DI has ROI calculators and specific metric promises. Perioskoup could add a simple **ROI framing** to the ForDentists page: "Clinics using patient engagement tools see X% fewer missed appointments" (cite source).

### Category Creation Assessment
The "AI dental companion" category framing is the right strategic move ‚Äî there is near-zero search competition. The site correctly frames against "chatbot / PMS plugin / fitness tracker" in `ForDentists.tsx:280-295`. This competitive differentiation copy is strong.

**Missing:** A comparison table or simple 3-column "Perioskoup vs. PMS vs. Patient Portal" would crystallize the positioning, especially for clinic decision-makers. This is the most common conversion tool on B2B SaaS landing pages.

## Summary

The Perioskoup landing page is well above average for a seed-stage health-tech product. The brand system is disciplined, the dual-audience architecture is clear, and the EFP award is deployed effectively. The 11-day gap to launch is tight but achievable if the critical issues ‚Äî especially **the missing form backend** and **Dongle font verification** ‚Äî are resolved first. The animation system is purposeful, not gratuitous, with the `prefers-reduced-motion` gap being the main accessibility concern. Competitive positioning is sharp; the category creation play ("AI dental companion") is the right strategic bet.
